## Context

当前应用已经完成第一阶段地基：新建/打开作品、草稿本和正文本、手动保存、重新打开后内容保留。下一步还不是 AI 写作交互，而是先让一个 LLM 配置真实可用、可测试，这样后续 AI 功能才有可信的后端调用路径。

需要沿用的现有代码模式：

- Rust Tauri commands 位于 `src-tauri/src/lib.rs`，并委托给领域模块处理具体逻辑。
- 前端调用后端的薄封装位于 `src/project-api.ts`。
- 面向用户的校验和异步状态处理已经出现在 `src/new-project-form.ts` 和 `src/editor.ts`。
- Rust 集成测试位于 `src-tauri/tests/project_test.rs`，通过临时文件系统状态测试真实生产代码。
- 用户文本边界必须继续保护：草稿本和正文本只由用户明确写入/保存流程改变。

Tauri v2 可以通过 `app_handle.path().app_local_data_dir()` / `app.path().app_local_data_dir()` 获取应用本地数据目录。LLM 配置属于应用级设置，不属于某个具体作品文件夹，因此应放在应用本地数据目录下。

## Goals / Non-Goals

**Goals:**

- 提供一个手动 LLM 配置：API 地址、API Key、模型名。
- 将该配置作为应用级系统配置保存和加载。
- 在任何 AI 写作 UI 出现之前，先用一次真实请求测试该模型配置是否可调用。
- 显示清楚的保存/测试状态和可读错误。
- 保持项目边界：LLM 配置不写入草稿本、正文本，也不写入 `project.json` 的用户元信息正文内容。

**Non-Goals:**

- 不做 AI 面板。
- 不做基于选区的“召唤 AI”。
- 不做提示词组装器、AI 回复历史或对话状态。
- 不做多个 provider、多个模型槽位，也不做前端显示但后端无法真实调用的模型列表。
- 暂不做系统安全钥匙串；本 change 可以用本地明文 JSON 保存 API Key，但必须在界面明确提示。
- 不做任何 AI 生成文本对草稿本或正文本的插入、替换、改写、移动或删除。

## Decisions

### 1. LLM 配置保存为应用级配置，而不是作品数据

LLM 配置应是整个桌面应用的全局设置，不绑定到某一个作品文件夹。实现时应新增一个小型 Rust 领域模块，例如 `llm_config`，负责解析应用本地数据目录，并在其中保存一个 JSON 配置文件。

原因：

- 模型连接是工具设置，不是某一部作品的内容。
- API Key 不应该出现在用户可见的 `作品文本/` 文件夹中。
- 应用级配置可以避免用户每新建一个作品就重新填写同一套 Key。

考虑过的替代方案：把配置保存在每个作品文件夹的 `next-story-system/` 下。这个方案能复用现有作品存储模式，但会让 API Key 变成作品级数据，容易和作品内容混淆，因此本 change 不采用。

### 2. 第一版只支持一种安全的 OpenAI-compatible chat-completions 配置形态

第一版真实可调用形态应为：

- API 地址：OpenAI-compatible base URL，例如某个服务的 `/v1` endpoint，不包含 `/chat/completions`。
- API Key：作为 `Authorization: Bearer <key>` 发送。
- 模型名：作为请求中的 `model` 字段发送。

API 地址使用结构化 URL 解析。远程地址必须使用 HTTPS；HTTP 只允许 `localhost`、`127.0.0.1` 和 `::1` 本机回环服务。地址不得包含用户信息、query、fragment 或完整 `/chat/completions` 路径。

测试连接使用异步 HTTP client，禁用自动重定向，向配置的 API 基础地址追加 `/chat/completions` 并发送一次最小请求。成功不仅要求 HTTP 2xx，还必须解析合法 JSON，并确认至少有一个非空 OpenAI-compatible `choices` 结果。

原因：

- 这是很多当前模型服务共同支持的最小实用形态。
- 它避免过早引入 provider 分支 UI，也避免假模型列表。
- 它直接服务项目铁律：UI 里能选/能填的模型配置，后端必须真实能跑。

考虑过的替代方案：允许任意自定义 HTTP endpoint。这个方案更灵活，但会立刻带来请求/响应格式、认证方式和错误处理的复杂性，因此不进入第一个 LLM 配置 change。

### 3. Rust 暴露加载、保存、测试三个命令

新增三个 Tauri commands：

- 加载已保存的 LLM 配置。
- 校验后保存 LLM 配置。
- 使用给定或已保存的 LLM 配置发起一次真实测试调用。

前端应通过类型化薄封装调用这些命令，延续现有 `project-api.ts` 模式。Rust command 层保持轻薄，把校验、文件读写和异步 HTTP 逻辑委托给新的领域模块。

原因：

- 前端不直接接触文件系统细节。
- 后续 AI 调用可以复用同一条已校验的后端配置路径。
- 符合当前架构：Rust 负责本地规则和文件能力边界。

### 4. 使用现有表单/状态模式添加最小配置视图

UI 应提供一个简单 LLM 配置视图，并能从现有应用壳进入。它包含：

- API 地址输入框。
- API Key 密码输入框。
- 模型名输入框。
- 明文保存风险提示。
- 保存配置动作。
- 测试连接动作。
- 成功/失败状态文本。

表单应复用当前模式：明确 DOM ID、集中 DOM 查询、本地校验后再调用命令、异步动作期间禁用按钮、短错误信息就地显示。

配置视图记录进入来源：从欢迎页进入则返回欢迎页；从编辑器进入则返回原编辑器，不销毁当前作品、当前本子或未保存文本。表单有效状态和异步忙碌状态分别管理；保存/测试期间输入变化不得重新启用动作按钮。加载配置使用递增 generation，过期结果不得覆盖较新的加载或用户已经修改的字段。

原因：

- 用户现在就能实际填写配置。
- 不需要在这个阶段发明庞大的设置系统。
- 保存/测试状态与现有新建作品、保存作品流程保持一致。

### 5. API Key 展示要谨慎，但不能假装已经安全存储

API Key 输入框应使用 password 类型。应用可以在重新加载后把已保存 Key 放回输入框，便于编辑和测试，但必须明确提示：开发阶段该 Key 是本地明文保存。

原因：

- password 输入框能降低旁人直接看到 Key 的概率。
- 明确提示能避免用户误以为当前已经接入系统安全钥匙串。
- 系统安全存储应在核心 AI 链路验证后，作为后续独立 hardening change 处理。

## Risks / Trade-offs

- API Key 明文保存 → 缓解：界面明确提示风险，并把安全钥匙串集成留给后续单独强化。
- provider 兼容性被误解 → 缓解：将 API 地址标注为 OpenAI-compatible，并用真实测试调用验证，而不是承诺广泛 provider 支持。
- API Key 通过远程 HTTP 泄露 → 缓解：远程地址只允许 HTTPS，本机回环服务可使用 HTTP，HTTP client 禁止自动重定向。
- 2xx 假成功 → 缓解：解析 JSON 并要求至少一个合法非空 `choices` 结果。
- 网络/测试失败可能返回技术化错误 → 缓解：将常见情况映射为可读信息：缺少字段、URL 无效、认证失败、模型不存在/不可用、网络失败、provider 响应异常。
- 设置视图可能膨胀成过早的设置系统 → 缓解：只做一个聚焦的 LLM 配置视图，不加入无关设置。
- HTTP 依赖会增加 Rust 表面积 → 缓解：只有需要时添加一个维护良好的 HTTP client，将它隔离在 LLM 配置领域模块，并尽量用测试/模拟覆盖。
- 配置导航打断未保存稿件 → 缓解：记录进入来源并返回原页面，不销毁编辑器内存状态。
- 并发操作或过期加载覆盖状态 → 缓解：独立 busy 状态、operation/generation 标识和 dirty 检查。

## Migration Plan

这是一个增量变更。现有作品和笔记本不需要迁移。

变更后第一次启动时，如果配置文件不存在，应用返回空配置状态。用户保存后创建应用级配置文件。回滚时可以忽略或手动删除该配置文件，不影响任何作品。
