## Why

第一个地基 change 已经证明：用户可以新建作品、打开作品、编辑草稿本/正文本、手动保存，并在重新打开后保留内容。下一步最稳妥的是先让一个真实 LLM 配置可以被后端调用，再继续做选区召唤 AI，避免重蹈旧项目“前端显示一堆模型，但后端实际跑不通”的问题。

## What Changes

- 添加一个单一 LLM 配置界面，让用户手动填写：
  - API 地址
  - API Key
  - 模型名
- 将这个唯一 LLM 配置保存在本地系统区域，与用户的草稿本和正文本分离。
- 应用启动或重新打开时加载已保存的配置。
- 添加“测试连接”动作，使用当前或已保存配置发起一次真实模型调用。
- 为测试连接显示清楚的成功/失败状态。
- 本 change 只包含配置和连通性验证。
- 不添加 AI 面板对话、选区召唤 UI、提示词生成、回复历史、多 provider、多模型槽位，也不添加任何写入草稿本或正文本的动作。

## Capabilities

### New Capabilities
- `llm-configuration`: 覆盖单模型 API 手动配置、本地持久化、加载已保存配置，以及验证该模型配置是否真实可调用。

### Modified Capabilities
- 无。

## Impact

- 前端：添加一个简单的 LLM 配置入口，包含表单状态、校验提示、保存/加载行为和测试连接状态。
- Rust 核心：添加本地系统配置数据结构，以及保存、加载、测试单一 LLM 配置的 Tauri commands。
- 项目/系统数据：LLM 配置保存到应用系统区域，不写入 `作品文本/草稿本.txt`、`作品文本/正文本.txt`，也不写入 `project.json` 的用户元信息正文内容。
- 依赖：如果当前 Tauri/Rust 栈没有合适的 HTTP 调用能力，可以添加一个 Rust HTTP client 依赖。
- 安全：开发期允许 API Key 以本地明文 JSON 保存，但界面必须明确提示风险；系统安全存储留给后续单独升级。
