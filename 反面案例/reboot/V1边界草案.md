# V1 边界草案

> 本文档是新项目重启后的 V1 范围草案。
> 它不是最终实现计划，而是防止 V1 再次膨胀的边界文件。

## 1. V1 的一句话目标

V1 只证明一件事：

```text
用户可以稳定地写剧本、保存剧本、重新打开剧本，并且系统不打扰他继续写。
```

如果某个功能不服务这句话，它默认不进 V1。

## 2. V1 的主链路

V1 只围绕这条链路：

```text
打开应用
  -> 创建项目
  -> 进入剧本编辑器
  -> 写 screenplay
  -> 保存 revision
  -> 刷新或重新打开
  -> 内容仍然存在
  -> 导出文本
```

这条链路稳定前，不做复杂 AI 能力。

## 3. V1 的核心对象

### 3.1 Project

项目是用户的一部作品容器。

V1 project 只需要：

- id
- name
- created_at
- updated_at

不需要：

- 多用户
- 权限
- 云同步
- 项目模板
- 项目级复杂偏好

### 3.2 Screenplay

剧本正文是 V1 的核心。

V1 只保留一个正文概念：

```text
screenplay
```

不再有 `main`。

### 3.3 Revision

revision 是正文版本记录。

V1 revision 只需要：

- revision_id
- project_id
- content
- created_at

可以后续再加：

- message
- author
- checksum
- diff
- branch

但 V1 不需要。

### 3.4 Chat

V1 可以有 Chat，但它只做主动求助。

Chat 内容默认是对话，不是事实源。

AI 回复默认是 suggestion，不是 screenplay。

用户点击采纳后，才进入 screenplay。

## 4. V1 的数据原则

V1 不追求最终完美架构，只追求不混乱。

### 4.1 唯一正文事实源

```text
screenplay revisions = 正文事实源
```

当前正文可以有缓存文件，但缓存必须能从 revisions 重建。

### 4.2 AI 输出不是事实源

```text
AI suggestion != project fact
AI summary != project fact
AI classification != project fact
```

用户采纳前，AI 内容不能改变正文或结构状态。

### 4.3 投影后置

V1 可以没有复杂 `story_state`。

如果需要当前状态，只做最小派生：

- 当前 screenplay 内容
- revision 列表

五模块投影、事实提取、矛盾检测都不进 V1。

## 5. V1 要做什么

### 5.1 项目管理

V1 支持：

- 创建项目
- 查看项目列表
- 打开项目

V1 不支持：

- 删除项目
- 复制项目
- 项目模板
- 多用户协作

删除是破坏性操作，等数据模型稳定后再做。

### 5.2 剧本编辑器

V1 支持：

- 输入正文
- 保存正文
- 重新打开正文
- 查看保存状态
- 基础导出文本

V1 不强求：

- 完整 Fountain 解析
- 场景导航
- 格式校验
- 多文档
- 分支版本树

### 5.3 Revision 记录

V1 支持：

- 每次保存生成 revision
- 读取最新 revision
- 简单查看 revision 列表

V1 不支持：

- 自动 diff
- 自动合并
- 分支
- 回滚 UI

### 5.4 Chat 主动求助

V1 可以支持一个极简 Chat：

- 用户主动输入问题。
- AI 回复。
- 用户可以手动复制或点击“插入正文”。

V1 Chat 不做：

- 自动提取事实
- 自动分类五模块
- 自动生成 idea cards
- 自动写记忆
- 自动触发矛盾检测

## 6. V1 明确不做什么

以下能力全部不进 V1：

1. 多 Agent 架构。
2. Thin Hub 调度系统。
3. Timing Policy。
4. 五模块自动分类。
5. 矛盾检测。
6. 涟漪检查。
7. 顽固模式。
8. 心流自动识别。
9. 用户长期记忆。
10. idea cards 自动提取。
11. LightRAG。
12. 多 LLM provider。
13. 双文档 `main` / `screenplay`。
14. 自动格式门禁。
15. 版本分叉和合并。

这些能力可以作为未来路线，但不能偷偷进入 V1。

## 7. V1 的 AI 边界

V1 中 AI 只能在用户主动请求时出现。

允许：

- 用户问：“帮我想想下一场。”
- 用户问：“这段对白有没有别的写法？”
- 用户问：“给我三个方向。”
- 用户点击“插入正文”。

禁止：

- 用户保存正文后，AI 自动分析并写状态。
- 用户聊天后，AI 自动分类到模块。
- AI 自动判断矛盾并插队提醒。
- AI 自动把自己的总结写入项目事实。

## 8. V1 的 UI 边界

V1 UI 只需要：

1. 项目列表 / 创建项目。
2. 剧本编辑器。
3. 极简 Chat 面板。
4. 简单设置页。

不需要：

- 监控台。
- 五模块画布。
- 存档树。
- idea 仓库。
- 多面板复杂布局。

页面位置由 URL 管，不由 store 管。

## 9. V1 的验证标准

V1 完成必须通过以下手动或自动验证。

### 9.1 项目链路

```text
创建项目 -> 刷新页面 -> 项目仍存在 -> 打开成功
```

### 9.2 正文链路

```text
输入正文 -> 保存 -> 刷新页面 -> 正文一致
```

### 9.3 Revision 链路

```text
保存两次 -> revision 列表有两条 -> 最新正文正确
```

### 9.4 Chat 链路

```text
用户提问 -> AI 回复 -> 不自动改变正文
```

### 9.5 采纳链路

```text
AI 回复 -> 用户点击插入正文 -> 正文增加 -> 保存后可回读
```

## 10. 进入 V2 的条件

只有当以下全部成立，才考虑 V2：

- V1 主链路稳定。
- 端到端验证可以重复运行。
- 数据事实源没有歧义。
- AI 不会自动写入用户未确认的事实。
- 用户实际使用时没有明显打断感。

V2 可以考虑：

- 轻量正文摘要。
- 手动触发的结构整理。
- 用户确认后的 idea 保存。
- 基础 Fountain 辅助。

仍不建议立即进入：

- 自动矛盾检测。
- 多 Agent。
- LightRAG。
- 复杂五模块自动同步。

## 11. V1 成功的定义

V1 成功不是“AI 很聪明”。

V1 成功是：

```text
用户愿意在里面继续写。
写下来的东西不会丢。
AI 不会越界。
系统不制造混乱。
```

如果这四件事成立，再谈智能。
