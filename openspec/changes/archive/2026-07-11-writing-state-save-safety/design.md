## Context

现有编辑器在 `src/editor.ts` 中已经维护当前作品、统一的 `hasUnsavedChanges` 标记、保存进行中状态和保存结果文案。草稿本与正文本分别由两个 `textarea` 持有，因此切换标签页时输入不会被主动清空；但这项行为尚未成为正式契约。当前作品切换入口和桌面窗口关闭也没有统一经过未保存修改检查。

作品文本只能通过现有 `save_project` Tauri command 写入草稿本和正文本文件。本 change 不改变这一事实源，不新增后台写入路径，也不把未保存内容另存为缓存、备份或版本。

## Goals / Non-Goals

**Goals:**

- 让草稿本与正文本在标签切换期间分别保留各自的未保存输入。
- 为所有会卸载当前作品的应用内操作提供统一的未保存修改保护。
- 为桌面窗口关闭提供同样的保护和三选一决策。
- 保证“保存并离开”只有在真实写盘成功后才离开；失败时不丢失当前输入。
- 统一保存状态与用户可见用语，并使状态转换可以独立测试。

**Non-Goals:**

- 不加入自动保存、自动备份、崩溃恢复或版本快照。
- 不改变草稿本、正文本和项目元数据的文件结构。
- 不加入 AI 面板、模型生成、选区处理或任何 AI 写入能力。
- 不重做编辑器布局、视觉样式或通用路由系统。
- 不保证操作系统强制终止进程、断电或崩溃时能够提示或恢复。

## Decisions

### 1. 用“当前内存内容与最后成功保存内容的比较”判断未保存修改

编辑器为草稿本和正文本各保留一份最后成功保存的内容。任一本子的当前内存内容与对应基线不同，整个作品就处于“有未保存修改”状态。每次保存成功后，同时更新两份基线；保存失败时不更新。

选择内容比较而不是只依赖一次性布尔标记，是因为用户可能把修改手动撤回到原内容。此时界面应重新显示“已保存”，而不是永久保持未保存状态。两个本子共享一套判断规则，避免状态逻辑分裂。

### 2. 标签切换不属于“离开作品”

草稿本与正文本始终各自由自己的编辑控件保留当前值。切换标签只改变当前可见本子，不触发保存、不弹提示，也不覆盖另一本子的内存内容。

备选方案是在每次切换前提示保存，但这会频繁打断写作，并且用户已经明确要求切回来时内容仍在，因此不采用。

### 3. 所有离开作品的入口共用一个异步离开守卫

应用内任何会替换或卸载当前作品的动作，都先向统一守卫提交“保存成功后要继续执行的动作”。守卫按下列顺序处理：

1. 没有当前作品或没有未保存修改：直接继续。
2. 存在未保存修改：显示应用内三选一提示。
3. 选择“保存并离开”：调用现有保存入口，成功后继续；失败则停留。
4. 选择“不保存并离开”：不写盘，直接继续。
5. 选择“取消”：关闭提示并停留。

这里的“离开作品”包括打开另一作品、返回欢迎页，以及今后新增但会卸载当前作品的入口。采用统一守卫而不是在每个按钮里复制判断，能确保三选一语义一致。

### 4. 桌面窗口关闭复用同一决策语义，但由关闭拦截器发起

应用监听 Tauri 窗口的关闭请求。若存在未保存修改，先阻止本次默认关闭，再进入同一三选一流程：保存成功或明确不保存后，由应用发起一次受控关闭；取消或保存失败则保持窗口打开。

受控关闭必须带有一次性“允许关闭”状态，避免应用再次发起关闭时重复触发提示。该状态只允许绕过紧接着的一次已确认关闭，不能长期关闭保护。

三选一使用应用自有对话框，而不是依赖只有“确认/取消”的二选一接口。这样按钮文字和行为在 Windows 上保持明确，也便于自动测试核心决策逻辑。

### 5. 保存过程只有一个进行中的 Promise

手动保存、“保存并离开”和快捷键保存调用同一个保存动作。保存进行中时不启动第二次保存；需要等待保存结果的离开流程等待同一个结果。保存动作返回明确的成功或失败结果，而不是只更新界面文字。

保存开始时显示“正在保存…”，成功后显示“已保存”，失败后显示“保存失败：<原因>”。用户继续输入后显示“有未保存修改”。为保持现有界面用语，本 change 沿用“有未保存修改”，不再引入“脏状态”等第二名称。

### 6. 保存时使用本次提交的内容快照

保存开始时冻结草稿本和正文本的本次提交内容。写盘成功后，只把该快照设为最后成功保存基线。如果用户在保存等待期间继续输入，新输入与保存基线仍不同，界面继续显示“有未保存修改”，不能错误显示“已保存”。

如果“保存并离开”期间允许用户继续输入，会产生“保存完成但最新输入未写盘”的歧义。因此三选一提示和后续保存进行期间暂时阻止编辑及重复离开动作；保存失败后恢复编辑，且所有输入保持不变。

## Risks / Trade-offs

- **[Risk]** 浏览器式 `beforeunload` 与 Tauri 窗口关闭行为不同，单靠网页事件可能漏拦截。 → **Mitigation:** 使用 Tauri 窗口关闭请求监听，并在真实桌面窗口中手动验证。
- **[Risk]** 受控关闭再次触发关闭监听，形成重复提示或循环。 → **Mitigation:** 使用只消费一次的允许关闭状态，并为状态转换增加单元测试。
- **[Risk]** 保存等待期间发生新输入，导致界面错误认为所有内容已保存。 → **Mitigation:** 保存冻结双本快照，并在成功后重新比较当前内容与基线。
- **[Risk]** 各离开入口分别实现提示，未来出现行为差异。 → **Mitigation:** 所有应用内离开动作只能经过统一离开守卫。
- **[Trade-off]** 本 change 只保护正常的应用内离开和窗口关闭，不能覆盖断电、崩溃或系统强制结束进程。恢复这些情况需要未来单独讨论自动备份或崩溃恢复，不在本范围内。

## Migration Plan

1. 从现有 `ProjectState.hasUnsavedChanges` 迁移到可由当前内容与最后保存基线推导的状态，不改变作品文件格式。
2. 将现有手动保存改为返回可供离开守卫等待的明确结果，同时保留原有 Tauri `save_project` command。
3. 将现有和后续的离开作品入口接入统一守卫，再接入桌面窗口关闭监听。
4. 如实现出现问题，可回退前端状态与关闭拦截代码；磁盘上的作品文件和项目元数据无需迁移或回滚。

## Open Questions

无。三选一按钮、标签切换行为、保存失败保护和范围排除项均已由用户确认。
