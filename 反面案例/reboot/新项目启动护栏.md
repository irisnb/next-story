# 新项目启动护栏

> 本文档是新项目开工前的检查表。
> 任何 proposal、设计、实现开始前，都应先过一遍本文档。

## 1. 开工前必须确认的五件事

### 1.1 用户第一主链路是什么

必须能用一句话说清：

```text
用户打开应用后，最核心的一条路是什么？
```

V1 推荐答案：

```text
创建项目 -> 写剧本正文 -> 保存 -> 重新打开 -> 内容还在
```

如果当前设计不是围绕这条链路，就不应进入 V1。

### 1.2 唯一事实源是什么

必须明确：

- 哪些数据是用户确认后的事实？
- 哪些只是缓存？
- 哪些只是投影？
- 哪些只是 AI 草稿？

默认规则：

```text
用户正文是事实源。
AI 输出不是事实源。
投影可以删，可以重建。
缓存可以丢。
```

### 1.3 一个概念叫什么

每个核心概念必须只有一个名字。

V1 建议命名：

| 概念 | 名称 |
|------|------|
| 项目 | project |
| 剧本正文 | screenplay |
| 正文版本 | revision |
| 聊天 | chat |
| 用户采纳 | adoption |
| AI 草稿 | suggestion |

如果有人提出第二个名字，先暂停命名，不继续实现。

### 1.4 V1 不做什么

任何 V1 文档都必须有“不做什么”章节。

没有“不做什么”的设计，默认不完整。

### 1.5 如何验证

每个功能进入实现前，都必须能写出至少一条可重复验证路径。

例：

```text
输入一段正文 -> 保存 -> 刷新 -> 重新打开 -> 内容一致
```

说不出验证路径，就先不实现。

## 2. 禁止直接进入 V1 的能力

以下能力不是永远不做，但默认不得进入第一版实现。

### 2.1 多 Agent 自动调度

原因：旧项目已经证明，多 Agent 在主链路未稳定时会放大边界混乱。

允许条件：

- 主链路稳定。
- 后台任务状态可见。
- 每个 Agent 有单一职责和输入输出契约。
- 用户能关闭自动行为。

### 2.2 自动五模块分类

原因：它容易把 AI 判断写成项目事实。

允许条件：

- AI 输出默认进入草稿区。
- 用户确认后才进入结构状态。
- 能明确区分用户文本和系统分析。

### 2.3 自动矛盾提醒

原因：它最容易踩“系统评判创意”的红线。

允许条件：

- 只递证据，不下结论。
- 心流时不打断。
- 用户可关闭。
- 有误报降权机制。

### 2.4 idea card 自动提取

原因：它会把聊天内容自动沉淀为项目资料。

允许条件：

- 用户知道它发生了。
- 用户能撤回。
- 不写入长期事实源。
- 失败可见。

### 2.5 多 LLM provider

原因：旧项目已经出现前端允许、后端不支持、spec 与实现不一致。

允许条件：

- provider registry 与 UI 列表一致。
- 每个 provider 有测试。
- API key 安全语义明确。

### 2.6 LightRAG / 知识图谱

原因：它适合作为后期能力，不适合进入实时主链路。

允许条件：

- 用于灵感检索或低频报告。
- 不进入实时矛盾判断。
- 不把检索结果当成用户事实。

## 3. 进入实现前的文档检查

每个实现前必须至少有：

1. 目标：这次只解决什么？
2. 非目标：这次明确不做什么？
3. 事实源：会写哪些数据？谁说了算？
4. 用户确认点：哪些操作必须用户确认？
5. 失败状态：失败时用户能否知道？
6. 验证路径：怎么证明它真的工作？

如果缺任何一项，不进入实现。

## 4. AI 助手协作护栏

新项目中，AI 助手必须遵守：

1. 事实和推荐分开写。
2. 每个推荐必须说明放弃什么。
3. 不可逆决策前做决策快照。
4. 不因为“看起来简单”跳过设计。
5. 不在用户未确认时扩大范围。
6. 不把旧项目文档直接当新项目规范。

## 5. 设计评审问题

每次设计完成后，问这些问题：

### 5.1 是否背离初心

- 它会不会让系统显得比用户更懂故事？
- 它会不会主动评判用户创意？
- 它会不会打断用户写作？
- 它会不会让 AI 输出自动变成事实？

### 5.2 是否边界过大

- 这个 change 是否同时改三个以上核心模块？
- 是否包含“顺便支持以后”的内容？
- 是否把 V2/V3 能力塞进 V1？

### 5.3 是否数据不清

- 新增数据属于事实源、投影、缓存还是草稿？
- 它是否能从事实源重建？
- 用户能否撤回或修改？

### 5.4 是否可测

- 是否有最小验证路径？
- 是否有失败路径？
- 是否需要真实 LLM 才能验证？如果需要，是否有 fake provider？

## 6. 开工前停机条件

以下情况出现，必须暂停：

1. “这个以后可能用到，所以先做。”
2. “这个先写死，后面再抽象。”
3. “AI 先自动整理，用户不喜欢再删。”
4. “先不写测试，手动点点看。”
5. “这个字段暂时叫 main，后面再统一。”
6. “失败只写日志就行。”
7. “这个 change 虽然大，但一次做完更省事。”

这些话不是绝对错误，但它们是旧项目失控的早期信号。说出这些话时，必须停下来写清代价。

## 7. 最小开工许可

只有当下面全部为真，才允许开始实现：

- V1 主链路已写清。
- V1 非目标已写清。
- 核心概念命名已写清。
- 数据事实源已写清。
- AI 写入权限已写清。
- 至少一条端到端验证路径已写清。
- 用户已确认当前范围。

缺一项，就继续写文档，不写代码。
