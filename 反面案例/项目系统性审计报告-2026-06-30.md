# 项目系统性审计报告（2026-06-30）

> 本文档记录 2026-06-30 对项目进行的只读系统性审计结果。
> 目的不是立即修复，而是把当前问题、证据、风险级别和后续收敛顺序明确沉淀下来，避免继续在混乱边界上逐个补丁式修 bug。

## 0. 审计结论摘要

当前项目的问题不是单个 bug 过多，而是多层边界同时漂移：

```text
OpenSpec 变更边界漂移
  -> 数据真相源定义冲突
  -> main / screenplay 文档 ID 分裂
  -> LLM/Agent 副作用链过长
  -> 前端 router / store 双状态源
  -> 后端非原子写入与内存状态失效
  -> 修复一个流程时破坏另一个流程
```

总体判断：应先做战略收敛，再进入逐项修复。继续直接修局部 bug，会继续出现“修好 A 又冒出 B”的症状。

### P0 级根因

| 编号 | 问题 | 影响 |
|------|------|------|
| P0-1 | OpenSpec active changes 互相覆盖边界 | 实现目标不稳定，AI/开发者会按不同版本理解系统 |
| P0-2 | `event log`、`story_state`、MD 文档、JSON 辅助文件的真相源角色混乱 | 数据不一致，投影和实际文件脱节 |
| P0-3 | `main` / `screenplay` 文档 ID 已分叉，但抽取和投影没有贯通 | 事实来源错标，正文/剧本读写互相覆盖或隔离 |
| P0-4 | LLM/Agent 一次聊天触发多条后台副作用，失败只写日志 | 用户看到成功，后台可能部分失败，形成隐性状态漂移 |
| P0-5 | 后端部分关键写入不是原子事务 | 正文 revision、adoption event、idea cards 可能写一半 |
| P0-6 | 前端 router 与 Zustand view store 双状态源 | 页面 URL、可见视图、项目状态可能不一致 |

### 3.2 `add-dialogue-agent` 变成大总包

相关文件：

- `openspec/changes/add-dialogue-agent/proposal.md`
- `openspec/changes/add-dialogue-agent/design.md`

风险：它覆盖范围过大，已经不只是“添加 Dialogue Agent”，而是重定义写作主链路。由于任务仍是 0/47，继续推进前必须先拆边界。

### 3.3 `dual-llm-config` 内部矛盾

相关文件：

- `openspec/changes/dual-llm-config/design.md`
- `openspec/changes/dual-llm-config/specs/llm-config/spec.md`

矛盾：

- spec 要求 API Key 使用 Fernet 加密，并从 `FIRST_STORY_ENCRYPTION_KEY` 读取密钥。
- design 决策写成 `base64(api_key + ':' + project_id)`，明确说明这只是混淆，不是真加密。
- 实现也使用 base64 混淆。

这是规格、设计、实现三者不一致。后续必须先决定 V1 到底接受“混淆”还是必须做到“加密”。

### 3.4 `five-modules-refactor` 与全局真相源原则冲突

相关文件：

- `openspec/changes/five-modules-refactor/design.md`
- `AGENTS.md`

冲突：

- `AGENTS.md` 明确：append-only event log 是真相源，`story_state.json` 是可重建投影。
- `five-modules-refactor/design.md` 把五模块 MD 文档描述为用户可见真相源，事件日志变成派生数据。

这是硬冲突。必须重新定义：模块 MD 是事实源、用户可见文档、还是投影/缓存。

### 3.5 `fix-chat-history-persistence` 已过窄

相关文件：

- `openspec/changes/fix-chat-history-persistence/proposal.md`
- `openspec/changes/fix-chat-history-persistence/design.md`

该 change 的聊天历史模型主要考虑：

- `role`
- `content`
- `message_id`
- `timestamp`

但后续系统已经引入：

- `script_ready`
- `intent`
- `extraction_status`
- `turn_count`
- idea card 自动提炼

所以该 change 即使完成剩余 endpoint 测试，也仍可能落后于当前真实语义。

### 3.6 OpenSpec 层建议

先不要继续开新功能。建议先做一个“战略收敛 change”，专门整理：

1. 哪个 change 是聊天主线。
2. 哪个 change 是 LLM 配置基础设施。
3. 哪个 change 是五模块文档结构。
4. 哪些 completed changes 需要归档。
5. 哪些 in-progress changes 需要并入、拆分、冻结或废弃。

相关文件：

### 4.4 模块编辑锁存在生命周期风险

相关文件：

- `frontend/src/features/modules/ModulesView.tsx`

问题：

- React effect 的依赖和清理逻辑存在 lint warning。
- 若 cleanup 捕获旧 lock 状态，可能导致真实锁未释放。

这与后端模块锁“每请求新建服务实例”的问题叠加，会让模块编辑表现为偶发残锁、锁失效或保存乱序。

### 4.5 Chat runtime 状态泄漏风险

相关文件：

- `frontend/src/features/chat/chat-runtime-context.tsx`
- `frontend/src/features/chat/ChatPanel.tsx`

问题：

- `scriptReadyMap` 是模块级状态，未按项目清理。
- 发送消息后存在延迟 invalidate，没有取消机制。
- `ChatPanel.tsx` 对 idea card 部分使用 raw fetch，不走统一 API client。

结果：项目切换、消息切换或请求返回顺序变化时，UI 可能显示旧状态。

### 4.6 idea card API 表面不统一

相关文件：

- `frontend/src/features/ideas/IdeaWarehouseView.tsx`

- feature 内部直接 raw fetch。
- 错误处理绕过统一 `ApiError`。

结果：后端契约变化时，类型系统不容易捕获，错误也不容易定位。

## 5. 后端审计

### 5.1 模块服务状态每请求重建

相关文件：

- `backend/app/services/project.py`
- `backend/app/services/module_document.py`

问题：

- `ProjectService.get_module_document_service()` 每次创建新的 `ModuleDocumentService`。
- `ModuleDocumentService` 的 locks、queues 都是实例内存状态。

结果：

- 不同请求之间不共享锁。
- queued additions 可能丢失。
- 系统以为有模块锁，实际锁只存在于当前 service 实例里。

这是模块编辑和后台分类写模块文档的核心风险。

### 5.2 adoption 链路不是完整原子事务

相关文件：

- `backend/app/api/chat.py`
- `backend/app/services/document.py`
- `backend/app/services/event_log.py`

流程：

1. `adopt_into_manuscript()` 读取当前 revisions。
2. 追加内容。
3. `DocumentService.save_revision()` 写入 `document.revised`。
4. 再写 `manuscript.adopted` 事件。

风险：

- 如果 revision 写入成功，但 adoption event 写失败，则正文已经改变，但幂等标记不存在。
- 同一个 `adopt_request_id` 重试时，可能再次追加内容。

### 5.3 idea cards 双写源

相关文件：

- `backend/app/api/chat.py`
- `backend/app/api/idea_cards.py`

问题：

- chat 后台 idea extraction 直接读写 `idea_cards.json` 和 `idea_card_revisions.json`。
- idea_cards API 也直接读写同一批 JSON 文件。
- 没有文件锁、没有原子 rename、没有统一 service 抽象。

结果：

- 后台任务和用户操作并发时可能互相覆盖。
- JSON 可能损坏。
- 前端看到的卡片状态可能和实际 revision 不一致。

### 5.4 projector 支持的事件没有 producer

相关文件：

- `backend/app/services/projector.py`
- `backend/app/services/module_document.py`
- `backend/app/models/events.py`

问题：

- `ProjectorService` 支持 `module_document.updated`。
- `models/events.py` 也定义了相关事件类型。
- 但模块保存路径 `ModuleDocumentService.save_module()` 只是直接写 Markdown 文件，没有 append 对应事件。

结果：模块文档内容可能已经变化，但 `story_state.modules` 不会通过事件流更新。

### 5.5 projector 错误吞噬与重建风险

相关文件：

- `backend/app/services/projector.py`
- `backend/app/api/state.py`

问题：

- `ProjectorService.load_state()` 对任意异常返回 `None`。
- `/state` 遇到 `None` 时会 rebuild。

结果：真实的投影损坏、schema 漂移、解析错误可能被表现成“状态偶发重建/状态忽隐忽现”，根因不容易暴露。

## 6. 持久化与真相源审计

### 6.1 当前存在三套持久化平面

```text
事件日志 JSONL
  - 设计上是真相源

story_state.json
  - 设计上是可重建投影

直接写盘辅助文件
  - project.json
  - modules/*.md
  - idea_cards.json
  - idea_card_revisions.json
  - llm_config.json
```

风险：这些平面没有统一事务。某个平面写成功、另一个失败，就会出现状态分裂。

### 6.2 `main` / `screenplay` 文档 ID 分裂

相关文件：

- `frontend/src/features/screenplay/ScreenplayView.tsx`
- `frontend/src/features/ideas/ManuscriptView.tsx`
- `frontend/src/features/chat/ChatPanel.tsx`
- `backend/app/api/chat.py`
- `backend/app/services/extraction.py`
- `backend/app/services/extraction_pipeline.py`
- `backend/app/models/state.py`

现状：

- `ScreenplayView` 读写 `screenplay`。
- `ManuscriptView` 读写 `main`。
- `ChatPanel` 采纳时显式写 `screenplay`。
- `AdoptRequest.document_id` 默认仍是 `main`。
- `ExtractionService.extract()` 默认 `document_id="main"`。
- `run_extraction_pipeline()` 没有 `document_id` 参数。
- `StoryState.current_document` 只有一个槽位，不按 document_id 分桶。

结果：正文/剧本文档语义不清，事实来源会错标为 `main`，投影也无法同时表达 `main` 和 `screenplay`。

### 6.3 event log append-only 但不是 crash-safe

相关文件：

- `backend/app/services/event_log.py`

问题：

- 事件写入是普通 append。
- 没有 fsync。
- 没有临时文件原子替换。
- 没有半行 JSON 修复策略。

结果：进程崩溃或写入中断可能损坏 JSONL，下一次读取/重放时直接失败。

### 6.4 `story_state` 校验不足

相关文件：

- `backend/app/services/projector.py`
- `backend/app/models/state.py`

问题：

- `story_state.json` 原子写回是优点。
- 但加载时主要依靠 `log_head_seq < event_count` 判断是否过期。
- `source_document_revision` / `source_document_checksum` 等字段存在，但未见完整赋值链路。

结果：投影是否对应正确文档/正确 revision 的证据不足。

## 7. LLM / Agent 审计

### 7.1 后台审计任务异常

原 `LLM agent audit` 后台任务异常：

- 原 session 在 `grep/glob` 后遇到 `Bad Gateway`。
- fallback session 只有用户 prompt 和空 assistant 消息。
- 没有工具调用、没有 transcript、没有结果。
- 已取消卡死任务，改为手动只读补审计。

### 7.2 provider 支持声明与实际实现不一致

相关文件：

- `frontend/src/shared/lib/llm-config-defaults.ts`
- `frontend/src/shared/api/api-types.ts`
- `backend/app/models/llm_config.py`
- `backend/app/services/llm_provider.py`

现状：

- 前端允许 `deepseek`、`jdcloud`、`openai`、`anthropic`、`local`。
- 后端 Pydantic 类型也允许这些 provider。
- 实际 `_PROVIDER_REGISTRY` 只注册 `deepseek` 和 `jdcloud`。

结果：用户可保存 OpenAI / Anthropic / local 配置，但真实调用时会失败。

### 7.3 API Key “加密”语义不一致

相关文件：

- `backend/app/services/llm_config.py`
- `backend/app/models/llm_config.py`
- `openspec/changes/dual-llm-config/design.md`
- `openspec/changes/dual-llm-config/specs/llm-config/spec.md`

问题：

- model 字段说明写“encrypted in storage”。
- spec 要求 Fernet。
- design 和实现使用 base64 混淆。

这是安全语义漂移。后续必须统一措辞与行为。

### 7.4 `get_provider_for_slot()` 不使用传入 project_service 定位项目

相关文件：

- `backend/app/services/llm_provider.py`
- `backend/app/services/project.py`

问题：

- 函数接受 `project_service` 参数。
- 但实际项目路径直接用 `settings.projects_root / project_id`。

结果：测试环境、自定义项目根目录、多 workspace 或 future migration 场景可能取错配置。

### 7.5 同一条 chat candidate 触发多个后台副作用

相关文件：

- `backend/app/api/chat.py`
- `backend/app/services/dialogue.py`
- `backend/app/services/extraction_pipeline.py`
- `backend/app/services/classify.py`
- `backend/app/services/context_summary.py`

流程：

一条 candidate 消息可能触发：

1. candidate extraction
2. module classification
3. idea extraction
4. minor/major context summary update

这些任务大多失败隔离，只写日志不返回给前端。

结果：用户看到 chat 成功，但后台可能部分写入、部分失败，形成不可见漂移。

### 7.6 candidate / committed 边界被打穿

相关文件：

- `backend/app/services/extraction.py`
- `backend/app/services/projector.py`

注释要求：candidate 模式只是 idea record，不应写 committed-world entity events。

实际风险：LLM semantic extraction 收集到角色名后，即使 candidate 模式也会写 `character.created`，只是在 payload 中带 `acceptance_status`。

如果 projector 没有严格按 acceptance_status 隔离，聊天里的临时想法会污染结构化世界状态。

### 7.7 分类入口使用不同 LLM 配置路径

相关文件：

- `backend/app/api/classify.py`
- `backend/app/api/chat.py`

问题：

- `/classify` API 直接从环境变量 `get_provider()` 取旧配置。
- chat 后台分类通过 `get_provider_for_slot(project_id, "utility")` 取项目配置。

结果：同一个分类能力，在不同入口下使用不同模型来源。

### 7.8 测试已落后当前 API contract

相关文件：

- `backend/tests/test_chat_api.py`
- `backend/tests/test_llm_config.py`
- `backend/tests/test_llm_provider.py`

问题：

- `test_chat_api.py` 仍断言 chat response 字段只有 `reply`、`message_id`、`intent`、`extraction_status`。
- 当前 `ChatResponse` 已包含 `script_ready`。
- `test_llm_config.py` 接受 base64 混淆，但 OpenSpec spec 要求 Fernet。
- `test_llm_provider.py` 只覆盖 deepseek provider，不覆盖前端声明的 openai/anthropic/local。

结果：测试无法真实保护当前规格。

7. 创意仓库新建/编辑/暂放/归档/删除/发送给 ChatUI。

## 9. 风险矩阵

| 风险 | 等级 | 证据 | 建议处理顺序 |
|------|------|------|--------------|
| OpenSpec 变更互相覆盖 | P0 | 多个 active changes 争同一主链路 | 第 1 |
| 真相源定义冲突 | P0 | AGENTS vs five-modules-refactor | 第 2 |
| `main` / `screenplay` 分裂 | P0 | 前后端 document_id 不一致 | 第 3 |
| LLM provider 声明与实现不一致 | P0 | 前端/模型允许 openai，registry 不支持 | 第 4 |
| Chat candidate 多副作用链 | P0 | chat.py 同时排 extraction/classify/idea/summary | 第 5 |
| 模块锁/队列内存态失效 | P0 | ModuleDocumentService 每请求新建 | 第 6 |
| adoption 非原子事务 | P1 | revision 与 adoption event 分步写 | 第 7 |
| idea cards 双写源 | P1 | chat.py 与 idea_cards.py 直接写同 JSON | 第 8 |
| 前端 router/store 双状态源 | P1 | URL -> store 单向同步，store 不导航 | 第 9 |
| `/manuscript` 死路由 | P1 | HomeView route 不存在 | 第 10 |
| 测试体系过期 | P1 | lint fail、pytest missing、contract test stale | 第 11 |

## 10. 建议修复战略

### 阶段 0：冻结扩张

在完成战略收敛前，不建议继续添加新功能。

允许做：

- 文档整理
- OpenSpec 边界重划
- 测试/验证环境恢复
- 明确阻塞 bug 的最小修复

不建议做：

- 新 Agent 能力
- 新 LLM provider
- 新 UI 模块
- 新数据结构扩张

### 阶段 1：OpenSpec 收敛

目标：让变更图重新可信。

动作：

1. 归档已完成且与当前主线一致的 change。
2. 冻结或拆分 `add-dialogue-agent`。
3. 修正 `dual-llm-config` 的加密/混淆决策。
4. 重写 `five-modules-refactor` 的真相源定义。
5. 校准 `fix-chat-history-persistence` 的消息 schema。

### 阶段 2：数据真相源收敛

目标：决定哪些数据是 source of truth。

必须回答：

1. event log 是否仍是唯一真相源？
2. module Markdown 是用户文档还是真相源？
3. idea cards 是否进入 event log？
4. `story_state.json` 是否只允许由 projector 写？
5. `main` 和 `screenplay` 是一个概念还是两个文档？

### 阶段 3：文档 ID 统一

目标：消除 `main` / `screenplay` 分裂。

可选路线：

| 路线 | 含义 | 放弃什么 |
|------|------|----------|
| 单文档路线 | 统一只保留 `screenplay` 作为正文 | 放弃 `main` 备用语义 |
| 双文档路线 | `main` 和 `screenplay` 都合法 | 必须重构投影、抽取、导出、UI |

推荐：先走单文档路线。理由是当前产品核心是剧本正文，双文档会显著扩大修复面。

### 阶段 4：LLM/Agent 副作用收敛

目标：一条 chat 消息不能暗中触发不可观测的多路写入。

建议原则：

1. Dialogue 只负责用户可见回复和 chat.message。
2. candidate extraction、classification、idea extraction、summary update 必须有清晰队列状态。
3. 后台任务失败要能被前端或日志索引追踪，不应只丢一条 warning。
4. candidate 不应写 committed-world 状态。

### 阶段 5：前端状态源收敛

目标：路由成为唯一页面状态源，或 store 成为唯一页面状态源，二者只能选一个主。

推荐：路由为主，store 只保存 UI 偏好和临时状态。

### 阶段 6：测试护栏恢复

目标：修复前先能稳定复现和验证。

动作：

1. 恢复 pytest 环境。
2. 修复前端 lint。
3. 更新结构契约测试路径。
4. 接入最小 Playwright e2e script。
5. 为 P0 修复建立回归测试。

## 11. 后续建议的 OpenSpec 工作

建议新建一个专门 change，例如：

```text
stabilize-system-boundaries
```

目标不是实现新功能，而是收敛系统边界。

建议包含：

- proposal：说明为什么当前需要战略收敛。
- design：定义真相源、文档 ID、LLM 副作用、前端状态源。
- tasks：先文档/测试，后最小代码修复。
- specs：只更新需要明确行为的能力，不一次性重写全部。

## 12. 决策快照

下面这些是后续不可继续含糊的问题：

1. `main` 和 `screenplay` 是否保留两个文档？
2. 五模块 MD 文档是否是真相源？
3. idea cards 是否进入 event log？
4. API key V1 到底是加密还是混淆？
5. 前端 provider 列表是否必须等于后端 registry？
6. candidate 是否允许创建 `character.created`？
7. Chat candidate 是否自动触发 classification 和 idea extraction？
8. Hub V1 是否明确只支持 single-process？
9. 测试门禁是否要求所有 change 必须有 e2e 或回归测试？

这些问题需要先定，再修。否则继续实现会把同一批矛盾搬到下一轮代码里。
