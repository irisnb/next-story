## Context

根目录 README 已经记录了当前命令、主要文件和已完成范围，但它先讲命令，再展开较长的文件树，最后才说明产品状态。目标读者是项目所有者和开发初学者，他们需要先知道项目是什么、当前做到哪里、什么尚未实现，以及哪些边界永久有效，然后才需要理解数据位置、技术路径和运行方式。

本次重写的事实依据分为四层。`AGENTS.md` 约束项目宪法和永久边界，`openspec/specs/` 只确认已实现能力，源代码与配置文件确认当前技术路径、数据位置和可运行命令，`方向/` 只能用于说明明确标注的未来方向。当前正式能力只有 `desktop-project-lifecycle`、`writing-notebooks` 和 `llm-configuration`。README 更新本身不改变这些能力。

## Goals / Non-Goals

**Goals:**

- 让读者按“项目身份 → 当前里程碑 → 已实现 → 未实现与永久边界 → 数据位置 → 架构与数据流 → 准备、运行和验证 → 文件索引 → 文档地图”的顺序理解项目。
- 只把正式规格支持的产品能力写成已实现，并保留现有 README 中经核对仍然准确的事实。
- 用大白话解释用户文本、系统元数据和应用级 LLM 配置的分离关系。
- 准确说明“前端 → bridge → Tauri commands → Rust domain”的单向调用结构，以及作品生命周期和 LLM 配置与连接测试两条数据流。
- 明确区分会执行完并退出的检查或构建命令，与需要用户主动停止的开发命令。
- 将项目宪法、已实现真相、未来方向和历史记录的文档职责分开说明。

**Non-Goals:**

- 不修改应用代码、产品行为、依赖、数据格式、构建配置或正式产品能力要求。
- 不修改现有正式规格、归档 change、`AGENTS.md`、`方向/` 或其他方向文档。
- 不把 AI 面板、选区召唤、备份、版本快照、多 provider 或多模型支持描述为已实现。
- 不设计或暗示任何 AI 直接写入、插入、替换、改写、删除或移动草稿本与正文本内容的入口。
- 不扩展为多份入门文档、架构文档或开发手册，只更新一份根目录 README。

## Decisions

### 1. 按读者问题排序，而不是按仓库结构排序

README 将先回答“这是什么”和“现在能做什么”，再回答“它怎么工作”和“怎么运行”。当前里程碑用简短状态卡集中呈现，已实现能力与未实现项目分开列出，永久 AI 边界单独强调。这样能避免初学者从命令或文件名反推产品状态。

备选方案是保留当前“命令 → 文件树 → 范围”的顺序，只补一段摘要。该方案改动较小，但核心信息仍然埋在后面，不能解决本次问题，因此不采用。

### 2. 已实现事实采用正式规格白名单

README 只有在 `desktop-project-lifecycle`、`writing-notebooks` 或 `llm-configuration` 中有正式要求支持时，才把产品能力标记为已实现。`方向/` 中的内容只能出现在明确标注为未来或未实现的部分，不能补充当前能力。源代码用于核对实现路径和文件位置，不替代正式规格成为产品范围来源。

备选方案是根据当前界面或源代码推断功能清单。这样可能把未归档、局部存在或仅预留的内容误写成已完成，因此不采用。

### 3. 将未来项目与永久禁止项分开

AI 面板、选区召唤、备份、版本快照、多 provider 和多模型支持等内容只能列为当前未实现，不能暗示已经可用。AI 不得直接改动草稿本和正文本则不是普通待办，而是永久边界。README 必须说明 AI 输出只能作为临时材料，只有用户亲手复制、粘贴、编辑并保存后，内容才进入作品事实。

备选方案是把所有内容放进一个“当前不包含”列表。该方案会把“以后可能做”和“永远不能做”混为一谈，削弱项目边界，因此不采用。

### 4. 用位置和责任解释三类数据

用户文本写入作品文件夹内的 `作品文本/草稿本.txt` 与 `作品文本/正文本.txt`。项目系统元数据写入同一作品文件夹内的 `next-story-system/project.json`，且不包含两个本子的正文。唯一应用级 LLM 配置写入 Tauri 提供的应用本地数据目录中的 `llm-config.json`，不写入作品正文或项目元数据正文内容。README 不猜测应用本地数据目录在各操作系统上的绝对路径，因为该路径由系统和 Tauri 决定。

备选方案是只列文件名，不解释责任与分离关系。该方案不能帮助读者判断哪些文件属于作品事实、哪些属于系统数据，因此不采用。

### 5. 架构说明使用一条主链和两条数据流

主链固定写为“前端 `src/` → bridge `src/project-api.ts` → Tauri commands `src-tauri/src/lib.rs` → Rust domain `src-tauri/src/project/` 或 `src-tauri/src/llm_config/`”。这里的单向是对一次请求的责任传递说明，结果和错误会沿调用链返回前端，不表示没有响应。

作品生命周期流覆盖新建、打开和手动保存。前端收集用户动作，bridge 调用对应 Tauri command，command 交给 Rust project domain 校验并读写作品目录，结果返回界面。LLM 流覆盖加载、保存和测试。前端配置表单经相同 bridge 调用 LLM commands，Rust llm_config domain 校验配置，将配置保存在应用本地数据目录，或使用当前唯一配置发出真实 OpenAI-compatible 测试请求，再把成功或可读错误返回界面。

备选方案是绘制完整模块图并逐文件解释。该方案会恢复当前 README 文件树过重的问题，也不利于初学者抓住责任边界，因此采用简短链路和两条流程。

### 6. 命令先说明前置条件，再按是否终止分组

README 将从仓库配置和 Tauri 开发环境要求核对前置条件，给出依赖安装步骤，然后列运行与验证命令。`npm run check`、`npm run typecheck`、`npm run test:frontend`、`npm run build`、`npm run test:rust` 和 `npm run tauri:build` 属于执行完成后返回的命令。`npm run dev` 与 `npm run tauri:dev` 属于持续运行、需要关闭窗口或按 `Ctrl+C` 停止的开发命令，并保留首次 Rust 编译可能较慢的说明。

备选方案是只给一个“快速开始”命令块。该方案容易让初学者把正常持续运行误判为卡死，因此不采用。

### 7. 文件索引只保留定位入口，文档地图单独说明真相层级

大文件树将压缩为帮助读者定位界面、bridge、Rust domain、测试和配置的简明索引，不逐项解释生成目录和锁文件。文档地图必须分别说明：`AGENTS.md` 是项目宪法，`openspec/specs/` 是已经实现的真相源，`方向/` 是未来方向，`openspec/changes/archive/` 中的归档 change 是变更历史，不代表当前入口或未来承诺。

备选方案是继续维护完整目录树。完整树容易过时，也会遮住读者真正需要的入口，因此不采用。

## Risks / Trade-offs

- [README 把未来方向误写成当前能力] → 对每条“已实现”声明逐项交叉核对三份正式规格，方向文档只允许出现在明确的未来标签下。
- [永久 AI 边界被弱化成暂未实现] → 为该边界设置独立章节，并在任务核对中明确检查 AI 永远不能写入两个本子。
- [架构说明为了易懂而失真] → 使用源代码中的真实文件和六个 command 核对主链与两条数据流，同时说明响应和错误会返回前端。
- [前置条件或命令随环境变化而过时] → 对照 `package.json`、Rust 配置和 Tauri 项目配置核对名称与用途，不写未经仓库或官方要求支持的版本号和绝对路径。
- [精简文件树后难以定位细节] → 保留按职责组织的简明文件索引，并让每个入口指向真实目录或文件。
- [文档 change 被误解为产品能力变化] → 只修改 `README.md`，并通过 OpenSpec 严格校验确认 delta 只约束项目说明，不改动三个现有产品能力。
