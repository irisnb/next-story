# 旧项目失败复盘

> 本文档复盘旧项目为什么失控。它不是责备记录，而是机制性教训。
> 目标是让新项目在类似信号刚出现时就停下来，不再等到 bug 连环爆发后才回头。

## 1. 一句话结论

旧项目失败的根因不是代码能力不足，而是：

```text
初心尚未被压缩成清晰 V1 边界
  -> 架构和机制提前膨胀
  -> 需求变化没有及时收敛
  -> 实现被多个互相冲突的规范拉扯
  -> bug 表现为连环爆发
```

## 2. 失败不是从代码开始的

旧项目在代码出问题之前，文档层已经出现了几个信号：

1. 机制设计越来越丰富，但 V1 边界没有同步缩窄。
2. Agent 架构越来越完整，但用户第一条主链路还不够稳。
3. OpenSpec changes 越来越多，但没有及时归档、合并、废弃。
4. 每个新想法都有道理，但没有一个明确规则决定“现在不做”。

这导致实现阶段出现一种假象：每个局部功能都说得通，但合在一起变成一个过早复杂的系统。

## 3. 失败链条

### 3.1 需求边界漂移

旧项目同时推进了多个 active changes：

- `add-dialogue-agent`
- `dual-llm-config`
- `five-modules-refactor`
- `fix-chat-history-persistence`
- `implement-minimal-writing-loop`
- `idea-cards-auto-extract`
- `fix-context-summary-trigger`

这些 change 各自合理，但它们都在触碰聊天主链路、LLM、事件日志、投影、文档保存、后台任务。

结果：

```text
每个 change 都以为自己只改一块
实际却在争同一条系统主干
```

### 3.2 真相源混乱

旧项目同时存在这些可能被当作事实源的东西：

- event log
- `story_state.json`
- 模块 Markdown 文件
- `project.json`
- `idea_cards.json`
- `idea_card_revisions.json`
- document revisions

原始理念是 event log 为真相源，`story_state.json` 为投影。但后来五模块 Markdown、idea cards JSON、文档 revision 都开始承担事实源角色。

问题不在“用了多个文件”，而在没有明确每个文件的角色：

| 文件类型 | 应该是什么 | 旧项目实际风险 |
|----------|------------|----------------|
| event log | 真相源 | 不是所有变更都进入它 |
| story_state | 投影 | 有时被当当前事实读 |
| Markdown 模块 | 用户可见整理文档 | 被设计成事实源候选 |
| idea cards JSON | 辅助数据 | 被多处直接写 |
| revisions | 正文版本 | 与 adoption event 非原子绑定 |

### 3.3 概念命名分裂

最明显例子是：

```text
main / screenplay
```

旧项目里：

- 剧本编辑器读取 `screenplay`。
- 另一个正文视图读取 `main`。
- Chat 采纳有时显式写 `screenplay`。
- 后端默认 document_id 仍是 `main`。
- 抽取管线默认来源也是 `main`。

这导致一个系统性问题：用户以为自己在写同一个正文，代码却在多个文档 ID 之间分裂。

### 3.4 AI 输出边界失控

旧项目原本有一个正确直觉：

```text
AI 输出默认不是事实，用户确认后才进入正文或状态。
```

但实现逐渐变成：

```text
用户发一条 chat
  -> Dialogue reply
  -> candidate extraction
  -> module classification
  -> idea extraction
  -> context summary update
```

这些后台动作有些会写事件，有些会写 Markdown，有些会写 JSON，有些只写日志。用户看到的只是一次聊天成功，系统内部却可能已经发生多路写入。

这违背了“用户判断权在用户手里”的初心。

### 3.5 前端状态源分裂

旧项目前端同时使用：

- URL route
- Zustand active view
- feature 内部局部状态
- TanStack Query cache
- 模块级变量

状态多不是问题，问题是没有明确谁是主。

表现包括：

- URL 能同步到 store。
- store 改变不一定同步到 URL。
- 某些按钮只改 store，不导航。
- 某些路径存在 UI 组件，但没有 route。

典型结果：页面看起来跳了，实际 URL 没跳；或者 URL 变了，状态还停在旧项目。

### 3.6 测试护栏滞后

旧项目有测试，但测试没跟上真实语义变化。

例子：

- Chat response 已增加 `script_ready`，测试仍断言旧字段集合。
- 前端结构测试还在找旧入口 `frontend/src/App.tsx`。
- LLM config spec 要求 Fernet，但测试验证的是 base64 roundtrip。
- 前端没有稳定 e2e script。
- 后端 pytest 环境在当前机器不可执行。

测试滞后后，系统没有能力阻止“修 A 坏 B”。

## 4. 关键误判

### 4.1 把“以后有用”误判成“现在该做”

LightRAG、五模块、矛盾检测、记忆、涟漪、Agent Hub 都可能有价值。

但旧项目的问题是：它们太早进入 V1 想象，导致第一条用户主链路尚未稳定时，系统已经开始背负复杂结构。

新项目必须区分：

```text
有价值 != 现在做
会用到 != V1 需要
设计过 != 已拍板
```

### 4.2 把“结构完整”误判成“系统可靠”

旧项目文档里有很完整的架构图和机制设计。但完整不等于稳定。

真正需要先验证的是：

- 用户能不能写。
- 保存后能不能回来。
- AI 会不会越界写状态。
- 一个概念是否只有一个名字。
- 一次操作是否只有一个可追踪结果。

### 4.3 把“失败隔离”误判成“失败可忽略”

旧项目很多后台任务失败后只写日志，不影响用户当前操作。

这对用户体验有好处，但如果没有可观测性，就会变成隐性损坏：

```text
前端显示成功
后台部分失败
状态已经不一致
用户几轮后才发现 bug
```

新项目要区分：

```text
不打断用户 != 不记录失败
失败隔离 != 失败消失
```

## 5. 早期警报信号

新项目中只要出现以下信号，就应该暂停设计或实现。

### 5.1 一个概念出现两个名字

例子：`main` / `screenplay`。

处理规则：停下来统一命名，不允许两个名字继续共存。

### 5.2 一个用户动作触发多个隐藏写入

例子：一次 chat 同时 extraction、classification、idea、summary。

处理规则：每个后台写入必须可见、可关、可追踪。

### 5.3 一个文件既是用户文档又是系统事实源

例子：模块 Markdown 既给用户看，又想作为状态事实源。

处理规则：用户文档、事实源、投影、缓存必须分角色。

### 5.4 一个 change 同时改三条主链路

例子：`add-dialogue-agent` 同时改 Hub、Chat、Extraction、Manuscript。

处理规则：拆 change。一个 change 只负责一个可验证目标。

### 5.5 测试开始解释不了真实行为

例子：测试断言旧 response contract。

处理规则：先更新测试和契约，再继续写功能。

## 6. 新项目必须吸取的教训

### 6.1 V1 要小到不能再小

旧项目最大教训是 V1 太早背负完整愿景。

新 V1 应该只做一条主链路：

```text
创建项目 -> 写剧本 -> 保存 -> 重新打开 -> 内容还在
```

### 6.2 AI 默认没有写入权

任何 AI 输出都只是建议、草稿或对话。

只有用户明确采纳，才能进入正文或结构状态。

### 6.3 后台任务必须晚于可观测性

如果系统还不能展示后台任务做了什么、失败在哪里，就不要让它自动写入项目状态。

### 6.4 文档先定“不要做什么”

旧项目文档大量写“可以做什么”。新项目文档要先写“V1 不做什么”。

### 6.5 旧文档只能作为证据，不是规范

旧文档中有些句子仍然对，但整套结构不能直接继承。

每次引用旧文档，都要标注：

- 保留
- 反例
- 待重新审问
- 废弃

## 7. 复盘结论

旧项目不是白费。它提供了非常清楚的反面证据：

1. 早期系统最怕过度聪明。
2. 多 Agent 之前先要有稳定主链路。
3. 数据事实源必须比功能更早定死。
4. 用户确认权必须编码到系统结构里。
5. 测试护栏必须跟着契约走，而不是跟着实现补。

新项目应从这些教训出发，先做一个克制、窄、能被验证的版本。
