# 旧项目概览

> 本文档是旧项目案卷入口。它不是新项目规范，也不是实现说明。
> 它只回答一个问题：这批旧材料是什么，为什么要保留，应该怎样阅读。

## 1. 旧项目是什么

旧项目原本想做一个剧本创作辅助工具。它的核心愿望不是替用户写故事，而是像一个剧作老师一样，在用户创作时提供陪伴、整理、提醒和启发。

旧项目的主要设想包括：

- ChatUI 作为主入口，用户可以自由聊天创作。
- 剧本编辑器承载正文写作。
- 五个模块承载结构化故事信息：世界、角色、情节、主题、结构。
- 后台 Agent 默默观察创作状态，必要时递证据，而不是替用户裁决。
- 事件日志作为真相源，投影文件用于快速读取。
- LightRAG 作为长期知识库方向。

这些设想里有一些仍然有价值，尤其是产品初心；也有一些被证明过早、过重、边界不清。

## 2. 旧项目为什么封存

旧项目不是因为某个单点 bug 被放弃，而是因为多个核心边界同时漂移：

```text
需求边界不稳
  -> OpenSpec changes 互相覆盖
  -> 数据真相源混乱
  -> main / screenplay 文档 ID 分裂
  -> LLM/Agent 副作用链过长
  -> 前端 router/store 双状态源
  -> 后端非原子写入与内存状态失效
```

继续在旧实现上修补，会把大量精力花在兼容旧错误上。由于当前进度仍是小测试阶段，选择重启比继续修补更稳。

## 3. 旧项目里仍值得保留的东西

### 3.1 产品初心

来自 `CLAUDE.md` 和部分架构文档的核心表达仍然成立：

- 创作没有唯一正确答案。
- 系统不得评判用户创意好坏。
- 理论是工具，不是答案。
- 用户比系统更懂自己的故事。
- 系统应递证据、给视角，而不是替用户下结论。

这些是新项目可以继承的初心材料。

### 3.2 反面经验

旧项目非常适合作为反面案例，尤其包括：

- 太早引入复杂 Agent 架构。
- 同一个概念出现多个名字。
- 用户发一条 chat 后，后台偷偷触发多条写入链路。
- 文档、事件日志、投影、模块文件同时承担事实源角色。
- OpenSpec change 没有及时归档和收敛。
- 测试护栏落后于实现变化。

这些经验应被写入新项目启动护栏，而不是被遗忘。

### 3.3 可借鉴的局部思路

旧项目里仍可借鉴的不是实现，而是局部思路：

- append-only 记录适合版本、生长、分叉。
- 用户主动求助和系统主动提醒必须区别对待。
- 后台发现应结构化，不应直接变成自然语言建议。
- 投影可以作为缓存，但不应取代事实源。
- 心流保护应高于纠错冲动。

这些思路需要在新项目里重新审问后再采用。

## 4. 旧项目里不应继续继承的东西

以下内容可以作为反例引用，但不应直接成为新项目规范：

- 当前 active OpenSpec changes。
- 旧实现代码结构。
- `main` / `screenplay` 双文档语义。
- 旧五模块自动分类链路。
- 旧 LLM provider 配置实现。
- 旧 Chat candidate 自动触发多后台任务的设计。
- 旧前端 router 与 Zustand view store 混合方式。
- 旧测试脚本和临时验证产物。

新项目只继承被重新确认过的东西。

## 5. 如何阅读旧项目材料

建议按下面顺序阅读：

1. 先读 `CLAUDE.md`，理解初心。
2. 再读 `docs/项目系统性审计报告-2026-06-30.md`，理解失败原因。
3. 再读本目录下的失败复盘和反面案例。
4. 最后再回看旧架构和机制设计。

不要反过来读。直接从旧架构开始，容易再次被复杂系统牵着走。

## 6. 旧项目的角色定位

旧项目从现在起被定位为：

```text
旧项目 = 反面案例 + 初心材料 + 证据库
旧项目 != 新项目规范
```

任何新设计若想继承旧项目内容，都必须先回答：

- 这继承的是初心，还是旧错误？
- 它是否会扩大 V1 边界？
- 它是否会引入第二个事实源？
- 它是否会让 AI 在用户未确认时写入项目状态？
- 它是否能被一个明确测试验证？

答不清，就先不继承。
