# 反面案例清单

> 本文档记录旧项目已经证明危险的模式。
> 新项目中一旦出现类似模式，应立即暂停，不要继续实现。

## 0. 使用规则

这不是普通建议，而是停机清单。

当新项目设计或实现中出现下面任何一种情况时，默认处理方式是：

```text
暂停 -> 写清事实 -> 重新命名/拆边界/缩小范围 -> 再继续
```

不能用“这次应该没事”绕过。

## 1. 同一个概念有两个名字

### 旧项目表现

`main` 和 `screenplay` 同时表示正文相关概念。

表现为：

- 前端剧本编辑器读写 `screenplay`。
- 另一个正文视图读写 `main`。
- 后端默认 document_id 是 `main`。
- Chat 采纳又显式传 `screenplay`。
- 抽取链路没有贯通真实 document_id。

### 为什么危险

同一个概念两个名字，会让每一层都以为自己是对的。最后出现：

```text
写入 A
读取 B
抽取标记 C
投影显示 D
```

### 新项目规则

一个产品概念只能有一个规范名。

V1 建议：

```text
剧本正文 = screenplay
版本 = revision
项目 = project
聊天 = chat
采纳 = adoption
```

如果出现第二个名字，先统一命名，再继续实现。

## 2. 一个用户动作触发多个隐藏写入

### 旧项目表现

用户发一条 chat，后台可能同时触发：

- candidate extraction
- module classification
- idea extraction
- context summary update

这些动作大多只在后端日志中可见。

### 为什么危险

用户以为自己只发了一句话，系统却在多个地方改状态。任何一个后台任务失败，都可能造成隐性不一致。

### 新项目规则

用户动作和写入结果必须一一可追踪。

V1 中，Chat 不自动写结构状态。

```text
AI 回复 = 草稿/建议
用户点击采纳 = 才写入正文
```

如果要新增后台写入，必须先回答：

- 谁触发？
- 写哪里？
- 用户是否知道？
- 用户是否能关闭？
- 失败如何显示？

答不清，不做。

## 3. AI 输出默认进入项目事实

### 旧项目表现

candidate extraction 可能写入角色、事实、模块内容，只是用字段标记 candidate。

### 为什么危险

创作早期想法很多都是试探。AI 自动把它们结构化，会把“聊聊看”变成“系统记住了”。

这违背初心：判断权在用户手里。

### 新项目规则

任何 AI 输出默认不是事实。

只有用户明确采纳后，才能进入：

- 剧本正文
- 结构化状态
- 长期记忆
- 项目资料

## 4. 文档、投影、缓存、事实源混在一起

### 旧项目表现

旧项目同时有：

- event log
- `story_state.json`
- module Markdown
- idea cards JSON
- document revisions

这些文件都在某些地方承担“当前事实”的角色。

### 为什么危险

当多个文件都像事实源时，系统无法回答：哪个数据说了算？

### 新项目规则

每种数据必须标明角色：

| 角色 | 含义 |
|------|------|
| 真相源 | 用户确认后的事实记录 |
| 投影 | 可删除、可重建的读取优化 |
| 缓存 | 可丢弃、可重新生成 |
| 用户文档 | 用户直接编辑的文本 |
| 草稿 | 未确认内容 |

任何新文件出现前，必须先写清它属于哪一类。

## 5. V1 塞进完整愿景

### 旧项目表现

V1 设计里同时出现：

- 多 Agent
- Thin Hub
- Timing Policy
- 五模块结构化
- 自动矛盾检测
- 心流模式
- LightRAG
- idea cards
- LLM 双配置
- 版本分叉

每个都有价值，但全部放进早期版本，导致主链路无法稳定。

### 为什么危险

愿景越完整，V1 越难验证。系统会在“看起来高级”之前先变得不可控。

### 新项目规则

V1 只做最小主链路。

```text
创建项目 -> 写 screenplay -> 保存 revision -> 重新打开 -> 导出
```

其他能力必须等这条链路稳定后再进入。

## 6. OpenSpec change 过大

### 旧项目表现

`add-dialogue-agent` 不只添加 Dialogue Agent，还覆盖 Hub、ChatUI、提取、采纳、记忆、事件。

### 为什么危险

change 过大时，任务列表看起来完整，实际无法判断每一步是否改变了系统边界。

### 新项目规则

一个 change 只能有一个主要目标。

好的 change 名称应该能回答：

```text
这次只改变什么？
```

如果一个 change 需要同时修改三个以上核心链路，先拆分。

## 7. 失败只写日志，不进入可观测状态

### 旧项目表现

多个后台任务失败时只 `logger.warning`，前端和用户无感。

### 为什么危险

这会制造“表面成功”。几轮后用户才发现数据没更新，定位困难。

### 新项目规则

失败可以不打断用户，但不能消失。

最低要求：

- 有任务状态。
- 有失败原因。
- 有可见位置。
- 有重试/忽略策略。

V1 如果没有任务状态系统，就不要做自动后台写入。

## 8. 前端 URL 和 store 同时管页面位置

### 旧项目表现

URL route 和 Zustand active view 同时表达当前页面。

有些地方导航 URL，有些地方只改 store。

### 为什么危险

页面状态不同步会制造非常难排查的 UI bug：看起来在一个页面，实际 URL、缓存、项目状态属于另一个上下文。

### 新项目规则

页面位置由 URL 管。

store 只能保存：

- 用户偏好
- 临时 UI 状态
- 草稿输入

store 不作为路由事实源。

## 9. 测试跟不上契约变化

### 旧项目表现

- Chat response 已变，测试还断言旧字段。
- 结构测试还找旧入口文件。
- LLM config spec 要求加密，测试验证混淆。

### 为什么危险

过期测试比没有测试更危险。它会给系统一种“有护栏”的错觉。

### 新项目规则

契约变化时，先更新测试，再更新实现。

如果测试无法解释当前行为，暂停功能开发。

## 10. 用户文档和系统分析互相覆盖

### 旧项目表现

五模块 Markdown 既像用户可见整理文档，又像系统结构化状态来源。

### 为什么危险

用户写给自己看的东西，和系统用来推理的东西，容错标准不同。

把它们合在一起，会让用户文本被系统规则污染。

### 新项目规则

用户可编辑文本和系统结构化分析必须分开。

系统可以从用户文本中生成分析，但分析不能反向覆盖用户文本，除非用户明确确认。

## 11. “先做聪明功能，之后再补边界”

### 旧项目表现

先实现自动提取、自动分类、自动卡片，再回头补状态边界、幂等、可观测性。

### 为什么危险

聪明功能最容易制造隐性副作用。边界后补通常补不干净。

### 新项目规则

顺序必须反过来：

```text
边界 -> 数据 -> 测试 -> 最小功能 -> 聪明功能
```

## 12. 触发停机的硬规则

以下任一情况出现，必须暂停：

1. 一个概念出现第二个名字。
2. 一个用户动作要写两个以上事实源。
3. AI 输出准备自动进入长期状态。
4. 一个 change 覆盖三个以上核心模块。
5. 测试描述和真实 API contract 不一致。
6. 新文件无法说明自己是真相源、投影、缓存还是草稿。
7. 后台任务失败后没有可见状态。
8. 前端某个状态同时由 URL 和 store 管。
9. V1 文档出现“顺便支持以后...”这类范围扩张。
10. 用户还没确认，系统就准备替用户整理、分类、修正创作内容。

## 13. 新项目默认反应

遇到反面案例时，不争论“这次会不会没事”。默认做：

```text
停
写清问题
缩小范围
重新命名
补测试
再继续
```

这份清单的价值不在于事后复盘，而在于事前刹车。
