## Context

当前“召唤 AI”链路以一次冻结选区快照为输入，经单请求协调器调用唯一保存的 LLM 配置，并在 AI 面板中使用 `replace-current` 策略显示单个结果。前端只发送选区原文，Rust 后端组装固定任务并发出一次非流式 `chat/completions` 请求。面板没有输入、消息轮次、历史或持久化。

本 change 要在不改变首次召唤低门槛入口的前提下，让用户在首次回应成功后继续追问。临时对话必须仍然锚定原冻结选区，不得因编辑器后续选区或文字变化而偷换依据；AI 输出仍是两个本子之外的临时材料。

## Goals / Non-Goals

**Goals:**

- 保持召唤时无输入：首次请求继续只使用冻结选区原文与固定任务。
- 首次回应成功后建立单条、线性、仅内存存在的临时对话，并开放追问输入。
- 继续追问时向模型提供原冻结选区与当前全部轮次，使模型能够理解此前交流。
- 让追问失败只影响当前待回答轮次，保留原选区、此前成功轮次与用户问题，并支持原样重试或修改后重发。
- 复用现有单请求、唯一 LLM 配置、非流式响应、作品隔离和零写回边界。

**Non-Goals:**

- 召唤时输入问题或文字。
- 多个临时对话、会话列表、历史、持久化、分支或消息版本。
- 自动摘要、截断、附近文本、全文、作品信息或 AI 内容库。
- 流式输出、停止生成、请求队列或并发追问。
- 修改此前成功的用户消息或 AI 回应。
- 思维扩展、思考收束或安全返回写作的完整交互。
- 任何插入、追加、替换、改写、删除、移动或整理草稿本与正文本的入口。

## Decisions

### 1. 前端持有一个线性临时对话，后端不保存会话

前端面板状态在首次成功后保存一个 `anchorSnapshot` 与有序轮次。用户轮次保存原始问题文本和状态，AI 轮次只保存成功回复；同一时刻最多有一个待回答用户轮次。后端每次只处理当前请求，不分配会话编号，也不保存消息。

选择这一方案，是因为当前应用已经由前端持有 AI 面板状态，作品加载与卸载也已有明确清理边界。后端持有会话会额外引入会话身份、清理、恢复与串话风险，而本 change 不需要这些能力。

### 2. 首次请求与追问请求使用结构化联合载荷

前端到 Tauri 的生成契约改为能够区分两类请求：

- 首次召唤：冻结选区原文，且没有用户问题或既有轮次。
- 继续追问：同一冻结选区原文，加上此前全部成功轮次与本次追问；本次追问只出现一次。

请求载荷使用显式、受限的消息角色结构，只允许 `user` 与 `assistant` 轮次；系统行为规则仍由 Rust 后端集中加入，前端不能提交或覆盖 system 消息。Rust 必须拒绝空白选区、空白追问、角色顺序非法、缺少首次 AI 回应或出现待回答问题之后仍有消息的载荷。

选择发送全部轮次，而不是后端会话或自动摘要，是为了让每次真实模型请求可检查、可复现，并避免摘要把用户问题或 AI 假设改写成事实。超过上游服务限制时返回现有安全错误，不偷偷摘要、裁剪或丢弃早期轮次。

### 3. 固定选区作为整个临时对话的不可变锚点

首次点击“召唤 AI”时冻结的本子类型、原文和位置继续属于当前临时对话。追问期间编辑文本、改变选区或切换本子，不改变锚点，也不改变已经成功的消息。

新的有效选区可以发起新召唤，但只有请求被协调器接受时才替换旧临时对话并显示新锚点的生成状态。旧请求的迟到结果通过现有请求 token 与新增对话 identity 双重隔离，不得进入新对话。

### 4. 一个协调器覆盖首次召唤与所有追问

首次请求和追问共用同一 `AiRequestCoordinator` 单请求锁。生成期间不能发起第二次召唤、追问或重试，不加入队列。编辑器仍可继续写作、切换本子和形成选区。

请求结果除了受现有作品 token 约束，还必须匹配发起时的临时对话 identity 与待回答轮次 identity。这样，即使用户在请求期间卸载作品或以后开始新对话，迟到结果也会被丢弃。

### 5. 面板从 replace-current 结果扩展为单个当前临时对话

首次生成之前和生成期间不显示追问输入。首次成功后，面板按时间顺序显示：首次 AI 回应、用户追问、对应 AI 回应。追问输入只接受至少一个非空白字符；发送后暂时禁用，直到当前请求结束。

面板收起不清空对话，重新展开恢复当前位置和内容；进入 LLM 配置页再返回也保留当前对话且不自动请求。返回欢迎页、卸载或替换作品、关闭应用时不持久化，现有内存状态随生命周期清空。

新召唤被接受时直接替换旧临时对话，不提示、不保留历史。该规则维持“只有一个当前 AI 工作上下文”的简单模型。

### 6. 追问失败是轮次状态，不是整个面板状态

提交追问后，用户问题立即成为当前待回答轮次。成功时追加 AI 回应并关闭该轮；失败时保留问题并在其后显示安全错误。用户可以：

- 原样重试：使用同一锚点、此前成功轮次和该问题重新请求；
- 修改后重发：只允许修改当前失败问题，再按新文本请求。

重试不会重新生成首次回应，也不会复制此前成功轮次。缺少配置时沿用相同失败轮次语义；前往配置页后返回不自动发送。

首次请求失败仍沿用现有快照级重试：面板尚未形成可追问临时对话，用户只能用原冻结选区重试首次请求。

### 7. 临时对话消息使用纯文本，保持零写回

用户问题和 AI 回应都按纯文本显示与发送；不解析模型返回的 Markdown 或 HTML。面板仍只提供标准复制，不提供插入、替换、应用、整理或保存两个本子字符的能力。追问输入只发送给模型，不是编辑器写入入口。

## Risks / Trade-offs

- **[完整轮次随请求增长，可能超过服务限制]** → 不自动摘要或截断；返回安全、可读的请求过长错误，并保留失败问题供用户修改或稍后重试。对话长度控制另开 change 讨论。
- **[前端消息结构可被错误组装]** → 前端使用受限类型，Rust 后端再次校验选区、角色顺序、空白内容和最后待回答轮次，不信任前端载荷。
- **[新召唤与旧追问结果串话]** → 每次新召唤建立新的对话 identity；结果同时校验作品 token、协调器请求 token、对话 identity 和轮次 identity。
- **[追问错误状态增加面板状态复杂度]** → 分开保存锚点、成功轮次、一个可选待回答轮次和请求状态；不引入多个会话或任意消息编辑。
- **[直接替换旧对话可能丢失用户想保留的临时材料]** → 这是用户已确认的最小范围取舍；结果始终可复制，但不在本 change 加历史或替换确认提示。

## Migration Plan

1. 先以测试锁定现有首次召唤、单请求、作品隔离和零写回行为。
2. 引入结构化生成载荷与 Rust 校验，同时保持首次请求组装出的模型消息与当前行为等价。
3. 将面板状态迁移为“一个锚点 + 线性轮次 + 可选失败轮次”，再接入追问输入和渲染。
4. 扩展生命周期、重试、迟到结果和 README 验证。
5. 若实现无法通过验证，可回滚本 change；作品文本和项目文件结构没有迁移，也没有新增持久数据需要清理。

## Open Questions

本 change 的产品边界已锁定，没有阻塞实现的开放问题。输入框具体视觉尺寸、消息间距和非关键文案可在既有设计语言内实现，并通过 UI 验收确认，不得借此扩大功能范围。
