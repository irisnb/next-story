# 第一版方向共识（2026-07-01）

> 本文档记录 2026-07-01 头脑风暴后，对新项目第一版方向达成的共识。
> 它不是实现计划，也不是最终规格。技术选型和交互细节尚未展开。
> 它只回答：这一版为什么做、做什么、明确不做什么、核心概念叫什么、数据谁说了算、怎么验证。
>
> 旧项目材料位于 `反面案例/`，仅作教训与反例参考，不作本项目规范。

## 0. 一句话定位

第一版是一个**提问式创作伙伴**：用户倒进碎片想法，AI 像一个懂剧作却只说人话的损友，不断递出能顶用户往前想的好问题，陪用户把故事一步步问出来。

## 1. 为什么做（Why）

### 1.1 真实痛点来源

方向来自一位真实创作者（动画创作者 / AI 影视从业者）的亲身剧本创作经历。核心痛点按重要性排序：

1. **想不出来**：这是最痛、最根本的问题。用户原话：想法最重要，有了想法再分门别类地整理不难；真正扯淡的是想不出来。
2. **灵感易逝**：碎片想法来得快、去得也快，需要被接住、不丢。
3. **经验门槛**：用户不能默认都受过剧作训练。有想法但缺方法论的人，同样要能被陪着把故事往前推。

### 1.2 灵魂原则

> **好问题 > 好答案。有了问题才有答案。**

AI 的核心价值是"问对问题"，不是"给对答案"。这一条同时天然满足旧项目拼命想补却补不上的安全边界：

- 提问不是结论 → 不替用户判断。
- 提问不产生"事实" → 不存在 AI 偷偷把答案写进项目。
- 提问把答案权留给用户 → 判断权 100% 在用户手里。

### 1.3 创作过程的真实机制（设计依据）

从用户的真实创作历程中提炼出的机制，是"问什么问题"的依据：

- 创作本质是**一场持续的自我提问**（"最开始的爆点是什么""两个人能发生什么""如何更有趣""新角色一定得是人吗"）。
- 创作往往**靠一轮轮对话被逼出来**（和同学聊了一轮又一轮，一点点往前推进）。
- 破局常靠**往旁边加一个维度**（两人太无趣 → 三角关系），但这只是解法之一，不是唯一解。
- 真实的高频死亡谷是"**有个炸点，但接不住**"（做爱开场定得很早，却一度接不下去）。

## 2. 第一版做什么（Scope）

### 2.1 AI 只做两件事

1. **问好问题**，覆盖四个卡点时刻：
   - ① 空白起头：面对空白，帮用户起个头。
   - ② 孤点接续：有个孤零零的点，帮用户往下接。
   - ③ 半路破局：走了一段堵住了，帮用户破局。
   - ④ 诊断加码：东西有了但不够，帮用户诊断、加码。
   - 可选：几个手动按钮，让用户主动指定当前想要哪一类问题。
2. **说人话**：不甩术语、不当裁判、不夸创意好、不给标准答案。
   - 知识（如麦基《故事》那套方法论）内化在 AI 里，门槛留在 AI 那边；用户只面对人话和好问题，不面对术语和理论。

### 2.1.1 AI 提问的知识来源（知识库的过渡方案）

AI 要"懂剧作才能问好问题"，但完整知识库（LightRAG）太重、太早（见第 3 节明确不做）。第一版用**最笨但最稳**的过渡方案：

- 先沉淀一份**麦基《故事》方法论的精炼 Markdown 文档**，作为 AI 提问的内功心法。
- 这份文档是**喂给 AI 的，不是给用户看的教科书**：AI 消化其中的方法（激励事件、进展纠葛、鸿沟、正反价值等），转化成人话问题吐给用户；用户永远看不到里面的术语。
- 等这个笨办法真的不够用了，再升级成知识库。这是同一需求（AI 懂剧作）从"轻"到"重"的自然演进路径。

### 2.2 界面两块

1. **对话区**：用户和 AI 一问一答想故事。
2. **想法收集箱**：用户**手动**把对话里的火花"钉"进去。这是收藏夹级别的一个按钮，不是自动整理模块。
   - **重要定位**：手动钉是**过渡方案**，不是最终形态。用户真正的痛点（灵感"瞬间到来、瞬间流逝"）理想解是**自动整理与收集**，那才是体验最好的。但自动整理要求"地基稳 + 可观测"，第一版不具备，且它超出本版唯一目标。所以第一版先用手动这个笨办法（先能用，再用好）。
   - **自动整理是明确的 V2 目标**，不是禁区。旧项目死于"边界漂移 + 多功能叠加"，不是死于"自动"二字。等第一版主链路稳了，自动整理正式提上日程。

### 2.3 LLM 接入

- 一个**通用配置框**：填 API Key + 模型名 + 接口地址（base URL），采用 OpenAI 兼容协议。
- 好处：多数国内外模型（豆包、DeepSeek、月之暗面等）都兼容该协议，用户填地址和模型名即可用，无需为每家单独写代码。
- 第一版先拿 **1~2 家**（如豆包 / DeepSeek）真实跑通验证。
- **单模型配置**，全局共用，不做双模型分裂。

## 3. 明确不做（Non-Goals）

以下能力不是永远不做，但**明确不进第一版**，记在案、不丢：

1. 多轨剧本编辑器 / 把混合态输出分轨归位（B 方向，后半段"呈现"问题）。
2. AI 自动整理、自动提取、自动分类。
3. 自动判断用户当前阶段（上个项目连"根据聊天记录自动整理"都炸了）。
4. 五模块结构化。
5. 知识库 / LightRAG（是未来核心壁垒，但不是前期地基；给 AI 用、人看不懂）。
6. 矛盾检测、心流自动识别、长期记忆、涟漪检查等。
7. 双文档 `main` / `screenplay` 语义。
8. 多 LLM provider 分裂配置（前端列一堆、后端不支持是旧项目 P0 雷）。

## 4. 核心概念命名（一个概念只能有一个名字）

| 概念 | 名称 |
|------|------|
| 用户和 AI 想故事的过程 | 对话 / chat |
| 用户倒进去的碎片想法与回答 | 想法 / idea |
| AI 递出的问题 | 问题 / question |
| 用户手动收藏的火花 | 收集箱条目 / pinned idea |
| 火花的存放处 | 想法收集箱 |
| 模型接入配置 | LLM 配置 |

（技术层命名待技术选型阶段再定，此处只锁产品概念。）

## 5. 数据事实源（谁说了算）

- **对话记录**是事实源：用户和 AI 说过的话，如实留存。
- **想法收集箱**是事实源：用户手动钉进去的火花。
- **AI 的问题和回复**不是"项目事实"，是对话内容本身；不自动转化为任何结构化状态。
- 整理动作**永远由用户手指发起**，AI 不自动写、不自动提、不自动分类。

## 6. 铁律（旧项目血的教训）

1. AI 绝不自动整理、自动提取、自动分类。
2. AI 输出永远是对话 / 建议，不是"项目事实"。
3. 整理动作永远由用户发起。
4. 第一版只服务"想不出来"这一个目标，不叠任何别的功能。
5. UI 里能选的每一个模型，后端必须真的能跑通。列表 = 实际支持。
6. 判断一个功能进不进版本的标准：**它是否服务本版唯一目标**，而不是"它是不是自动的"。自动本身不是原罪，旧项目死于"边界漂移 + 多功能叠加"，不是死于"自动"二字。

## 7. 技术形态（方向倾向，未最终拍板）

> **注意：技术栈目前是"暂选"，不是"已定"。** 用户是开发新手，技术地基必须"亲手碰过、确认不会卡在环境配置这第一步"之后才正式拍板。在那之前，以下都是倾向，不是铁板钉钉。若验证时发现 Electron 对新手太劝退，仍可换更简单的方案。

- 形态：本地桌面应用，能双击打开的独立窗口（不是浏览器网页）。**（这一条较确定）**
- 桌面框架：Electron。**（暂选，待亲手验证环境搭建是否顺利）**
- 语言：TypeScript。**（暂选，随桌面框架一起定）**
- 项目性质：开源 + 跨平台（Windows / macOS / Linux）。**（这一条较确定）**
- 必须包含：接入大语言模型 API。**（这一条确定）**
- 协作模式：代码主要由 AI 助手编写，用户负责产品判断与验收（用户是开发新手）。**（确定）**

## 8. 验证路径（怎么证明第一版成立）

第一版成功不是"AI 很聪明"，而是这几条能稳定跑通：

1. **配置链路**：填入 Key / 模型名 / 接口地址 → 保存 → 能成功调用一次模型。
2. **对话链路**：用户倒一句想法 → AI 递出一个人话好问题 → 一问一答能顺畅继续。
3. **收集链路**：对话中用户点"钉住"某条 → 该条进入想法收集箱 → 之后能回看。
4. **不越界验证**：AI 全程不自动整理、不自动分类、不产生项目事实；整理只发生在用户点击之后。
5. **换模型验证**：换一家 OpenAI 兼容模型（如豆包 ↔ DeepSeek），对话链路照样跑通。
