# Next Story

Next Story 是一个剧本写作桌面编辑器。当前版本只建立第一版地基：新建作品、打开作品、草稿本/正文本编辑、手动保存、重新打开后内容保留。

## 命令分两类：会自己停的，和不会自己停的

这一点非常重要，先看懂再运行，否则很容易把"正常运行"误判成"卡住"。

### 一、会自己跑完并返回的命令（测试 / 检查 / 打包）

这些命令有明确终点，跑完会自动返回，可以放心等它结束：

```bash
npm run check       # 一键：类型检查 + 前端构建 + Rust 测试，全绿才算通过
npm run typecheck   # 只查 TypeScript 类型
npm run build       # 只构建前端
npm run test:rust   # 只跑 Rust 测试
npm run tauri:build # 打出安装包（时间较长，但最终会返回）
```

日常想确认"代码没被改坏"，只需要跑 `npm run check` 一条即可。

### 二、不会自己停的命令（开发服务器 / 桌面预览）

这些命令是"一直开着"的，它们的工作就是持续运行，**不返回是正常的，不是卡死**：

```bash
npm run dev        # 前端开发服务器，一直开着实时刷新
npm run tauri:dev  # 桌面应用开发模式，一直开着让你操作窗口
```

它们像视频软件的"循环预览"：会一直放着，直到你主动关掉窗口或按 Ctrl+C 停止。
第一次运行 `tauri:dev` 需要先编译 Rust，可能要几分钟，属于正常等待。
运行期间它只在"有事发生时"打印日志（启动、你改了文件、报错），你不动它时它安静地开着，也属于正常。

**结论：想测代码用第一类命令；想亲手操作应用用第二类命令，并且知道它不会自己结束。**

## 项目文件结构导览（大白话）

这个项目由两半组成：**前端（你看到的界面）** 和 **后端（真正读写文件的 Rust 核心）**。
界面用网页技术写（HTML/CSS/TypeScript），底层能力用 Rust 写，Tauri 把两者打包成一个桌面软件。

下面按"重要程度"分三类讲，不是按字母顺序。

### 一、你真正在意的东西（你的作品）

```
src/                    前端源码（界面的大脑）
├── main.ts             启动入口：开机时把各模块接起来，很短
├── dom.ts              统一抓取页面上的按钮、输入框，一处管理
├── views.ts            管页面切换（欢迎页 ↔ 新建页 ↔ 编辑器）
├── new-project-form.ts 新建/打开作品的表单逻辑和校验
├── editor.ts           编辑器：草稿本/正文本标签、保存、保存状态
├── project-api.ts      前端和 Rust 后端对话的“电话总机”
├── types.ts            数据形状定义（项目长什么样、状态有哪些字段）
├── styles.css          所有界面样式（颜色、间距、字体、深色模式）
└── assets/             三个 logo 图标（脚手架自带）

src-tauri/              后端源码（真正碰你硬盘的那一半）
├── src/
│   ├── main.rs         程序总开关（启动那一下）
│   ├── lib.rs          对前端开放的三个命令：建作品 / 开作品 / 存作品
│   └── project/        核心业务
│       ├── mod.rs          数据结构定义 + 高层业务函数
│       ├── operations.rs   真正读写文件（建/存/开，含防损坏的临时文件保存）
│       └── validation.rs   校验规则（名字合不合法、位置能不能用）
├── tests/
│   └── project_test.rs 后端自动测试（调用真实生产代码）
├── capabilities/
│   └── default.json    权限清单：规定这个应用被允许干什么（如弹文件夹选择框）
├── tauri.conf.json     应用总配置：名字、窗口大小、图标、打包方式
├── build.rs            编译前准备脚本（Tauri 自动用，基本不碰）
├── Cargo.toml          Rust 版的“依赖 + 项目信息”
└── Cargo.lock          Rust 版的“依赖精确版本锁定”

openspec/               我们“怎么做事”的规范（分镜脚本）
├── changes/            正在进行的变更（要做什么、怎么设计、任务清单、规格）
└── specs/              真相源：已经做成了什么（归档后才有内容）

方向/                    产品蓝图：我们想做个什么东西（核心方向、第一版共识）
反面案例/                 旧项目的教训报告，作为“别再这么干”的反面参考
AGENTS.md               项目宪法：铁律和协作规则，你我开工前都先读它
README.md               本文件，项目说明书
```

### 二、根目录的配置文件（标准位置，看一眼知道是啥就行，平时不用碰）

这些文件放在根目录是全世界前端项目的通行标准，各种工具会自动来根目录找它们，**不要挪进子文件夹，否则会连环出错**。

```
package.json        前端项目的身份证 + 命令表（npm run 后面那些命令都在这里定义）
package-lock.json   依赖的精确版本锁定表（保证换台电脑装出来一模一样）
tsconfig.json       TypeScript 的检查规则设置
vite.config.ts      开发服务器（Vite）的配置
index.html          界面骨架：定义欢迎页、新建页、编辑器三个页面的结构
.gitignore          忽略清单：告诉版本控制哪些东西不要记录
```

### 三、工具自动生成的（完全无视，可删可重建，gitignore 已帮你藏起来）

这些不是你写的，是工具下载或生成的。删掉也没关系，用命令能重新生成。

```
node_modules/        前端下载的所有依赖零件（npm install 生成，几百个小包）
dist/                前端构建产物（npm run build 生成）
src-tauri/target/    Rust 编译产物 + 打出来的 exe / 安装包（几百 MB）
.git/                版本控制的历史仓库（Git 管，别手动碰）
.vscode/             VS Code 编辑器的项目设置
.omo/ .opencode/     AI 工具的运行记录和环境
```

### 快速对照：想改什么，去哪找

| 你要改… | 去哪 |
|---|---|
| 界面长相（颜色、间距） | `src/styles.css`、`index.html` |
| 界面行为（按钮、页面切换） | `src/` 里对应的 `.ts` |
| 文件读写 / 保存逻辑 | `src-tauri/src/project/` |
| 应用名字 / 窗口 / 权限 | `src-tauri/tauri.conf.json`、`src-tauri/capabilities/` |
| 加命令 / 看依赖 | `package.json`（前端）、`src-tauri/Cargo.toml`（后端） |
| 看项目规矩 | `AGENTS.md` |
| 看要做什么 | `openspec/changes/` |

## 第一版地基范围

这次地基变更**包含**：

- Tauri + TypeScript + Rust 桌面应用骨架
- 欢迎页（新建作品、打开作品）
- 作品文件夹结构：`作品文本/草稿本.txt`、`作品文本/正文本.txt`、`next-story-system/project.json`
- 纯文本的草稿本、正文本编辑标签页
- 手动保存，一次写入两个本子

这次地基变更**不包含**（属于后续变更）：

- AI 面板
- 大语言模型 / 模型配置
- 自动备份
- 版本快照
- AI 写入草稿本或正文本
